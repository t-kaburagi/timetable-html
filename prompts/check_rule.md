以下は、そのまま使える **LLMチェック用プロンプト雛形**です。
想定は「このツールから出力された `slot` 付きJSONをそのまま貼り付けてチェックする」運用です。
（ルールブックは別途あなたが同じスレッドに貼る／または固定で持たせる想定）

---

# prompts/timetable_check_prompt.md

## 目的

* UIで配置した時間割JSONを、**科目配置ルールブック**に照らしてチェックする。
* 本LLMは**判定者**。UIには戻らない。
* ただし「最終決定」は人間が行うので、**強制ではなく指摘＋修正案**を出す。

---

## 使い方

1. このプロンプトを先に貼る
2. 続けて「ルールブックの本文」または要点（A1/A2/A3…等）を貼る
3. 最後に UIの出力JSON（配列）を貼る
4. LLMの出力をもとに、必要箇所だけUIで手直し

---

## プロンプト（雛形）

あなたは大学の時間割編成を支援する「配置ルール監査官」です。
以下に与える **ルールブック** と **時間割JSON**（カード配置結果）を読み、違反・注意点・改善案を厳密に指摘してください。

### 前提（データ仕様）

* 入力はJSON配列。各要素は科目カードを表す。
* 主なフィールド：

  * `id`（必須）: 授業コード（例：ISC224a）
  * `title` : 授業名
  * `teacher` : 担当
  * `color` : 表示色（監査には原則不要）
  * `slot` : 配置先。`<period>/<day>` 形式。未配置は `null`。
* dayは `M, TU, W, TH, F`。
* periodは `1,2,3,L,4,5,6,7`（LはLunch）。
* 同一slotに複数カードが存在してもよい（別教員等）。ただしルール上の衝突は指摘してよい。
* “固定ブロック”（教授会、チャペル等）も通常カードと同じ形式で含まれる（灰色カードなど）。

### あなたのタスク

1. まず、JSONを解析し、次のサマリを作成：

   * 学期（もしJSON内にterm等があれば）/ 総カード数 / 未配置数
   * day×periodの占有状況（例：5/THに何件あるか）
   * 固定ブロックらしきカード（titleやidにBLOCK等を含むもの）の一覧

2. ルールブックに基づき、**違反（must）** と **注意（should）** を分類して列挙：

   * **違反（must）**：ルール上 “不可 / 禁止 / must not” に該当するもの
   * **注意（should）**：推奨違反、例外許容、運用上のリスク（学生負担・衝突可能性等）

3. 各指摘は、必ず以下を含める：

   * 対象カード（`id` と `title`）
   * 該当slot（例：`5/TH`）
   * どのルールに抵触しているか（ルール本文の該当箇所を短く引用 or 要約）
   * 具体的な修正案（可能なら代替slot候補を複数）

4. 代替案の作り方（重要）

   * 代替slotは、与えられたグリッド範囲（M..F, 1..7+L）の中から選ぶ。
   * 空きがない場合は「空きがない」と明示し、他カードの移動を含む提案にしてよい。
   * ただしあなたは最終決定者ではないため、複数案（A/B/C）を提示し、トレードオフを書く。

5. 最後に、修正案を反映した **提案JSON** を出力：

   * 元のJSON配列と同じ順序で出す。
   * 原則 `slot` のみ変更し、それ以外のフィールドは保持する。
   * 未配置のカードは `slot:null` のままでよい（ただし配置推奨があればコメント）。

### 出力フォーマット（厳守）

以下の順で、Markdownで出力してください。

## 1. Summary

* Total cards: ...
* Unplaced: ...
* Occupancy highlights: ...
* Suspected fixed blocks: ...

## 2. Violations (must-fix)

* [ID] Title — slot — Rule — Why — Fix options

## 3. Warnings (should-consider)

* [ID] Title — slot — Rule/Reason — Suggestion

## 4. Suggested changes (minimal-impact plan)

* Change list in bullet points (old slot -> new slot)

## 5. Proposed JSON (updated slots)

```json
[ ... ]
```

## 運用メモ（任意）

* ルールブックが長い場合は「must/should」を明記して要点化したものを貼るとブレが減ります。
* 固定ブロックを厳密に扱いたい場合、`id` を `BLOCK_*` で統一すると検出が安定します。
